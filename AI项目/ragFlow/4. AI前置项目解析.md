# RAGFlow 项目解析（面向 Java 开发者）

## 一、RAGFlow 是什么？

**RAGFlow** 是一个开源的 **RAG 引擎**，可以简单理解为：

- 帮你把 **自己的文档（Word、PDF、Excel 等）** 变成可查询的“知识库”
- 用户用自然语言提问
- 系统从文档中检索相关内容，再交给大模型生成回答，并给出引用来源

### 核心概念（和 Java 对照）

| 概念 | 通俗理解 | 类比 |
|------|----------|------|
| **RAG** | Retrieval-Augmented Generation，检索增强生成 | 相当于「先查 DB + 再生成」，类似：先查数据库拿到相关记录，再交给 LLM 生成回答 |
| **LLM** | 大语言模型（如 ChatGPT、通义千问） | 可类比为“会写文字的 API 服务” |
| **Embedding** | 把文本转成向量 | 类似把文字编码成高维数组，用于相似度检索 |
| **向量库** | 存储向量并做相似度搜索 | 类似 ES 的全文检索，但基于“语义”而不是关键词 |

## 五、核心流程（两大主流程）

### 流程 1：文档入库（类似 ETL）

```
上传文件 → 解析（PDF/Word...） → 切片（Chunk） → 向量化 → 存入 ES
         deepdoc/parser          rag/app/*        rag/llm/embedding
```

- `document_app.py` 接收上传
- `task_executor.py` 异步执行解析、切片、向量化
- 结果写入 Elasticsearch（文本 + 向量）

### 流程 2：问答（RAG 主链路）

```
用户问题 → 向量化 → 检索（ES） → 重排序 → 拼 Prompt → LLM 生成 → 返回答案+引用
           embedding   rag/nlp/search    rerank    prompt_config
```

- `conversation_app` / `api_app` 接收请求
- `dialog_service.chat()` 负责整体逻辑
- `retriever.retrieval()` → `Dealer.search()` 做检索
- 召回结果 + 用户问题 → 组装 prompt → 调用 LLM
- 答案和引用一起返回

## 七、Python 里和 Java 的几点差异

## 八、学习路径建议

1. **先跑通系统**：按 README 用 Docker Compose 起全量服务，走一遍“上传文档 → 建知识库 → 对话”流程。
2. **跟踪一个请求**：从前端提问开始，跟着 `conversation_app` → `dialog_service.chat()` → `retriever.retrieval()` → LLM 调用。
3. **理解 RAG 链路**：重点看 `api/db/services/dialog_service.py` 和 `rag/nlp/search.py`。
4. **再看文档处理**：`document_app` → `task_executor` → `deepdoc` 解析 → `rag/app` 切片。
