# RAGFlow 项目解析 —— Java 开发者视角

> 面向：有 Java 经验、未写过 Python 和 AI 项目的开发者

---

## 一、项目是什么？用一句话说

**RAGFlow 是一个「基于文档的智能问答」后端引擎**：你上传 Word/PDF/Excel 等文档，建「知识库」；用户提问时，系统从知识库里检索相关片段，再交给大模型生成带出处的回答。

- **RAG** = Retrieval-Augmented Generation（检索增强生成）  
  - 先**检索**：从你的文档里找到和问题相关的段落（chunks）  
  - 再**生成**：把检索到的内容 + 用户问题一起喂给 LLM，让 LLM 基于这些内容回答，并标注引用  
- 所以：**不是**纯聊天机器人，而是「你的文档 + 大模型」的组合系统。

## 二、和 Java 项目的对应关系（帮你快速建立直觉）

| 概念 | Java 里你熟悉的 | RAGFlow (Python) 里的大致对应 |
|------|------------------|--------------------------------|
| Web 框架 | Spring Boot | **Flask**（轻量 HTTP 服务） |
| 入口 | `Application.java` / `main` | `api/ragflow_server.py` |
| 接口层 | Controller | `api/apps/*_app.py`（如 `dialog_app.py`） |
| 业务逻辑 | Service | `api/db/services/*_service.py` |
| 数据模型/表 | JPA Entity、MyBatis | **Peewee**（ORM）+ `api/db/db_models.py` |
| 配置 | application.yml | `conf/`、`docker/service_conf.yaml.template` |
| 依赖管理 | Maven / Gradle | **pyproject.toml**（类似 pom.xml） |
| 文档解析 | 用 Apache POI 等 | **deepdoc/**（PDF/Word/Excel/PPT 解析 + OCR/版面分析） |
| 向量/检索引擎 | 可能用 Elasticsearch | **Elasticsearch** 或 **Infinity**（存向量+全文） |
| 外部大模型 | 调第三方 HTTP API | **rag/llm/** 里封装 OpenAI/国产大模型等 |

整体上：**API 层（Flask）+ 服务层（Service）+ 任务执行（异步解析/建索引）**，和 Java 里的分层思想一致，只是语言和生态是 Python。

## 四、核心流程 1：用户提问 → 回答（RAG 检索 + 生成）

可以理解为一条请求的「生命周期」：

1. **入口**  
   - 请求到 `api/apps/dialog_app.py` 的聊天接口，进 `DialogService.chat()`（在 `api/db/services/dialog_service.py`）。

2. **准备**  
   - 根据当前对话（dialog）确定：用哪个知识库、用哪个 Embedding 模型、用哪个 LLM。  
   - 从消息里取出最近几条「用户问题」`questions`。

3. **检索（RAG 的 R）**  
   - 调用 `retriever.retrieval(questions, embd_mdl, tenant_ids, kb_ids, ...)`。  
   - 内部会：  
     - 用 **Embedding 模型**把问题转成向量；  
     - 在 **ES/Infinity** 里做向量检索 + 全文检索（见 `rag/nlp/search.py` 的 `Dealer.search`）；  
     - 可选：Rerank、知识图谱、Tavily 联网等。  
   - 得到一堆「相关 chunk」：`kbinfos["chunks"]`。

4. **拼知识库上下文**  
   - 用 `kb_prompt(kbinfos, max_tokens)` 把 chunk 转成给 LLM 看的 `knowledge` 文本（会截断以控制 token）。

5. **生成（RAG 的 G）**  
   - 系统提示词里带 `{knowledge}`，和用户消息一起发给 **LLM**（`rag/llm/` 里封装的各家模型）。  
   - 若开启引用：再调用 `retriever.insert_citations(...)` 在回答里插入引用标记。

6. **返回**  
   - 流式或非流式返回 `answer` + `reference`（引用了哪些 chunk/文档）。

**对应代码位置**（便于你跳转）：

- 对话入口与参数：`api/db/services/dialog_service.py`（约 100–230 行：检索、拼 knowledge、调 LLM）。  
- 检索实现：`rag/nlp/search.py` 的 `Dealer.search`（向量+全文、过滤、排序）。

## 五、核心流程 2：文档上传 → 可被检索（解析 → 分块 → 建索引）

1. **上传与任务创建**  
   - 用户上传文件到某个知识库；后端创建解析任务（任务状态在 DB 里）。

2. **异步执行**  
   - `rag/svr/task_executor.py` 从 Redis 等队列里消费任务。

3. **解析**  
   - 按文件类型选解析器：PDF → `deepdoc/parser/pdf_parser.py`，再配合 `deepdoc/vision/`（OCR、版面、表格结构）；  
   - Word/Excel/PPT → `deepdoc/parser/` 下对应 parser。  
   - 输出：带结构的「文本 + 表格/图」等。

4. **分块（Chunk）**  
   - 按知识库配置的「解析模板」选策略：`rag/app/` 下如 `paper`、`laws`、`table`、`naive` 等。  
   - 把解析结果切成一块一块的 chunk（带页码、位置等元数据）。

5. **向量化 + 写入**  
   - 用 **Embedding 模型**把每个 chunk 转成向量；  
   - 和全文一起写入 **Elasticsearch** 或 **Infinity**，供上面「检索」那一步用。

**对应代码位置**：

- 任务执行与解析/分块入口：`rag/svr/task_executor.py`（FACTORY 里映射 parser 类型 → `rag.app` 下的模块）。  
- 文档解析：`deepdoc/parser/`、`deepdoc/vision/`。

## 六、关键概念（用 Java 思维理解）

- **知识库 (Knowledge Base / KB)**  
  - 类似「一个文档集合 + 一套配置」：用哪种解析方式、哪种 Embedding、哪个索引。

- **Chunk**  
  - 文档被切分后的「一段内容」；检索时找的是 chunk，不是整篇文档。

- **Embedding**  
  - 把一段文字变成一个向量（一串浮点数）；「相似」的文本向量接近，用于相似度检索。  
  - 代码里：`rag/llm/embedding_model.py` 等，调用各家 Embedding API 或本地模型。

- **LLM（大模型）**  
  - 负责最后「根据 knowledge + 问题」生成回答；配置在 `service_conf` 里，代码在 `rag/llm/chat_model.py` 等。

- **Rerank**  
  - 检索出一批 chunk 后，再用一个小模型对「问题-段落」相关度打分，排序后只留最相关的，减少噪声、省 token。

## 七、技术栈小结（便于你查资料）

- **语言**：Python 3.10+  
- **Web**：Flask  
- **ORM**：Peewee  
- **存储**：MySQL（业务数据）、Elasticsearch 或 Infinity（向量+全文）、Redis（队列/缓存）、MinIO（文件）  
- **AI 相关**：  
  - 文档解析/OCR：自研 deepdoc + 部分 ONNX  
  - Embedding/LLM/Rerank：通过 HTTP 调 OpenAI、国产大模型等（见 `rag/llm/`）  
- **依赖管理**：`pyproject.toml`，用 `uv` 安装（`uv sync`）。

## 八、建议的阅读顺序（按你的目标）

1. **只想会用、能部署**  
   - 看根目录 `README.md` + `docker/`，不深挖 Python 也行。

2. **想搞懂「问→答」**  
   - `api/apps/dialog_app.py` → `api/db/services/dialog_service.py`（chat 主流程）→ `rag/nlp/search.py`（Dealer.search）。

3. **想搞懂「文档怎么进知识库」**  
   - `api` 里文档/知识库相关接口 → `rag/svr/task_executor.py` → 选一个 `rag/app/xxx.py` 和 `deepdoc/parser/xxx_parser.py` 看一条链路。

4. **想改解析/分块策略**  
   - `rag/app/`（按文档类型）+ `deepdoc/parser/`、`deepdoc/vision/`。

这样你可以用「分层 + 数据流」的方式理解项目，和 Java 里的 Controller → Service → 外部系统 的思路是一致的，只是换成了 Python 和 RAG 术语。
