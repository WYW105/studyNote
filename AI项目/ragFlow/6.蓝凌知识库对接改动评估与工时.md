# 蓝凌知识库对接改动评估与工时

本文档基于《知识库系统与云从AI数据对接接口文档》及当前 RAGFlow 项目中飞书同步实现，评估「改用蓝凌知识库」所需改动与工时。

---

## 一、现有飞书数据流梳理

### 1.1 整体链路

```
飞书知识库 → 定时/手动触发 → 遍历节点(getFileList/traverse_nodes) 
  → 目录与文件元数据落库(feishu_documents_info) 
  → 需同步文件列表(need_sync=1/2) 
  → 按文件下载(Feishu_Operation.exportTask/checkExpResult/downLoadFile等) 
  → 写入 OSS/MinIO(oss_operation / minio_operation) 
  → oss_to_rag：从 OSS/MinIO 读文件 → FileService.upload_document / 富文本后端 
  → RAGFlow 解析/切片(parse_file_by_sdk 等) 
  → 更新 feishu_documents_info(doc_id, kb_id, is_sync 等)
```

### 1.2 关键模块与职责

| 模块/文件 | 职责 |
|-----------|------|
| **feishu_sync/auto_task.py** | 定时任务：每天 6:00 执行；`traverse_nodes` 全盘扫描 → `floder_to_oss` 更新目录 → `get_update_item_list_auto()` 取需同步列表 → 逐条 `feishu_file_to_rag`；飞书 token 定时刷新 |
| **feishu_sync/init_file_catalog.py** | `traverse_nodes` 递归遍历飞书节点；`floder_to_oss` 将目录结构写入 DB 与 OSS 路径；`knowledgebase_to_db` 初始化知识库元数据表；`checkFeishuFile` / `sqlOperation` 增删改 feishu_documents_info |
| **feishu_sync/sync_operation.py** | **核心**：`export_to_oss` 按类型( doc/docx/sheet/file 等) 调用飞书 API 导出/下载 → 写入 OSS/MinIO；`oss_to_rag` 从 OSS/MinIO 读文件 → 构建 FileStorage → `FileService.upload_document` + 富文本处理 → `parse_file_by_sdk`；`feishu_file_to_rag` 串联：先 export_to_oss，再删旧 doc，再 oss_to_rag；删除文档时 `delete_feishu_bucket_file` |
| **feishu_sync/feishu_operation.py** | 飞书 SDK：exportTask、checkExpResult、handleFile、downLoadFile、getSheetInfo 等，与飞书 Open API 直接交互 |
| **feishu_sync/get_node_filelist.py** | 调用飞书 API 获取节点/文件列表(getFileList) |
| **feishu_sync/get_user_access_token.py** | 飞书 OAuth：获取/刷新 user_access_token，供上述调用使用 |
| **feishu_sync/oss_operation.py** / **minio_operation.py** | 根据 `rag_storage.bucket_type` 选 OSS 或 MinIO，提供 put/get/rm 等，飞书文件统一写入同一 bucket（如 feishu/ 前缀） |
| **feishu_sync/mysql_executor + feishu.sql** | feishu_documents_info 表 CRUD：节点 token、路径、is_sync、need_sync、doc_id、kb_id、user_id 等 |
| **feishu_sync/task_manager.py** | 异步任务注册与 RQ 执行(`@async_task`)，如 `feishu_file_to_rag`、`knowledgebase_to_db` |
| **feishu_sync/call_rich_text_backend.py** | 对 pdf/ppt/doc 等走富文本解析流程，调用 DocumentService 等 |
| **api/apps/__init__.py** | `init_scheduler()` 启动飞书定时任务；`register_feishu_sync` 注册 feishu_sync 下蓝图 |
| **api/apps/document_app.py** | 删除文档时调用 `delete_feishu_bucket_file(doc_id)`，与飞书 bucket 清理联动 |
| **feishu_sync/configs_reload.py** | 配置从 Redis/本地 configs.yaml 读取：feishu_info、oss_info、minio_info、sdk_info、mysql、rag_storage 等 |

### 1.3 数据表与存储

- **feishu_documents_info**：飞书节点/文件元数据（space_id, node_token, obj_token, file_type, feishu_path, oss_path, is_sync, need_sync, doc_id, kb_id, user_id 等）。
- **存储**：OSS 或 MinIO 的同一 bucket，路径前缀如 `feishu/`，与 `rag_storage.bucket_type` 一致。

## 二、蓝凌接口与飞书流程的映射

| 飞书现有能力 | 蓝凌/文档接口 | 说明 |
|--------------|----------------|------|
| 定时全量/增量拉取文件列表 | **知识查询接口** POST queryKnowledge | 全量：不传 startDate/endDate；增量：传 startDate、endDate（格式 yyyy-MM-dd HH:mm）；按文档：传 fdId。替代飞书 traverse_nodes + getFileList。 |
| 按文件下载内容 | **附件下载链接获取接口** GET getDonwloadUrl?fdId={attId} | 知识查询返回 attId、attName；用 attId 换下载 URL，再 HTTP GET 下载文件流。替代飞书 exportTask/downLoadFile。 |
| 写入 RAGFlow | **保持不变** | 仍：下载 → 写入 MinIO/OSS → FileService.upload_document + 解析/切片；仅「数据来源」从飞书改为蓝凌。 |
| 变更感知（可选） | **知识库变更通知接口**（蓝凌 → 我方） | 文档新增/更新/失效时蓝凌回调我方；我方需提供 HTTP 接口接收，并触发「按 fdId 查询 + 下载 + 入 RAG」或标记待同步。 |
| 认证 | Basic Auth 或 国密 SM3 | 替代飞书 OAuth；请求头携带 Authorization(Basic) 或 Ekp-Rest-Date + Ekp-Rest-Auth(SM3)。 |

**结论**：

- **数据源切换**：不再调用飞书 API，改为调用蓝凌「知识查询」+「附件下载」；列表与文件内容均来自蓝凌。
- **存储与 RAG 流程**：继续使用现有 MinIO/OSS + `FileService.upload_document` + 解析/切片逻辑；可复用 `oss_to_rag` 中「从存储读文件 → 上传文档 → 解析」的主干，但需适配数据源为「蓝凌文档列表 + 蓝凌下载文件」。
- **元数据**：需新增或改造表结构，存储蓝凌侧字段：fdId、cateId、cateName、docSubject、docNumber、attId、attName、fdCreatorName、fdCreatorTime 等，并保留与 RAGFlow 的 doc_id、kb_id、user_id 映射。

### 2.1 蓝凌同步文档的 MinIO 位置与知识库绑定（重要）

**问：蓝凌同步过来的文档应该上传到 MinIO 哪个位置？知识库不是绑定 Agent 用的吗？**

- **MinIO 位置由「知识库（kb_id）」决定，不需要单独配置路径。**  
  RAGFlow 里上传文档时调用 `FileService.upload_document(kb, file_objs, user_id)`，内部使用 `STORAGE_IMPL.put(kb.id, location, blob)`：  
  - **第一个参数 `kb.id` 即 MinIO 的 bucket 名**（或等价于「存储位置」）；  
  - 第二个参数 `location` 为对象键（一般为文件名）。  
  因此：**只要在同步时传入要归属的 RAGFlow 知识库（kb_id），系统就会把文件写入该 kb 对应的 MinIO 位置**（bucket = kb_id）。无需再单独指定「MinIO 哪个目录」。

- **知识库确实用于绑定 Agent。**  
  知识库 = 一批文档的集合，Agent 绑定某个知识库后，用该库中的文档做 RAG 检索。因此：  
  **蓝凌同步前必须先确定「这批文档要进哪个 RAGFlow 知识库」**，即确定 `kb_id`（及 `user_id`/tenant）。同步后文档落在该知识库下，Agent 绑定该知识库即可检索到蓝凌同步的文档。

- **和飞书一致的用法**  
  飞书同步时，前端/接口传入 `user_info: { kb_id, user_id }`，表示「把选中的飞书文件同步到该知识库」。蓝凌同理：  
  - **方案 A（推荐）**：配置「蓝凌分类 → RAGFlow 知识库」映射（如 `cateId`/`cateName` → `kb_id`），同步时根据文档的 `cateId` 查映射得到 `kb_id`，再调用 `FileService.upload_document(kb, ...)`，文档自然落入对应 kb 的 MinIO 位置。  
  - **方案 B**：所有蓝凌文档同步到同一个 RAGFlow 知识库（固定一个 `kb_id`），由 Agent 绑定该知识库即可。

- **中间存储（可选）**  
  与飞书类似，可先把蓝凌附件下载到「中间 bucket/前缀」（如 `kms/`）再读入内存、调用 `upload_document`；最终落盘位置仍由 `kb_id` 决定（`kb_id` 对应的 bucket），中间路径仅用于同步流水线，不决定「文档归属哪个知识库」。

## 三、需改动/新增的模块与文件

### 3.1 新增：蓝凌（KMS）同步模块

建议新增目录 **`kms_sync/`**（或 `lanling_sync/`），与 `feishu_sync/` 并列，便于后续保留飞书或二选一。

| 文件/目录 | 说明 |
|-----------|------|
| **kms_sync/config.py** | 蓝凌 base_url（测试/生产）、认证方式（basic / sm3）、Basic 账号密码、SM3 key；时间格式等。可从 conf 或 Redis 读取，支持环境区分。 |
| **kms_sync/auth.py** | Basic Auth：拼接 `account;password` → Base64 → Authorization。国密 SM3：实现 signature = hmac_sm3(HttpMethod + URI + Date + sm3(HttpBody), key)，小写十六进制；请求头 Ekp-Rest-Date、Ekp-Rest-Auth。 |
| **kms_sync/client.py** | HTTP 客户端：封装「知识查询」POST queryKnowledge（分页、startDate/endDate、fdId）；「附件下载」GET getDonwloadUrl(fdId) → 再 GET 该 URL 下载 bytes。统一走 auth 模块加签/认证。 |
| **kms_sync/sync_operation.py** | 蓝凌版同步逻辑：根据「全量/增量/按 fdId」调用 client 拉列表；对每条记录用 attId 取下载 URL，下载文件流 → 写入 MinIO/OSS（路径建议 `kms/` 或可配置前缀）；构造与现有一致的「文件列表」结构，调用与 feishu 共用的「上传到 RAG」逻辑（见下）。处理「知识失效/退出」：删除或标记本地文档。 |
| **kms_sync/notify_receiver.py** | （若需要接收蓝凌变更通知）提供 HTTP 接口：接收 JSON（updateType、fdId、cateId、cateName）；校验后入队或写表，触发按 fdId 同步或标记失效。返回 code=200, msg="接收成功"。 |
| **kms_sync/task_manager.py** | 可复用 feishu_sync 的 Redis/RQ 机制，或独立队列；注册「蓝凌同步任务」等。 |
| **kms_sync/models/db 与 SQL** | 新建表如 `kms_documents_info`：fd_id, cate_id, cate_name, doc_subject, doc_number, att_id, att_name, oss_path, doc_id, kb_id, user_id, sync_time, update_type 等；或复用/扩展现有表并加 source=feishu/kms 区分。 |
| **kms_sync/logger_config.py** | 日志 logger，与 feishu_sync 类似。 |

### 3.2 复用与改造的现有能力

| 现有能力 | 改造方式 |
|----------|----------|
| **MinIO/OSS 读写** | 继续用 `feishu_sync/oss_operation`、`minio_operation`（或抽成通用 storage 层）；蓝凌文件写入单独前缀如 `kms/`，避免与 feishu 冲突。 |
| **上传到 RAG + 解析** | 抽离 `sync_operation.oss_to_rag` 中「从存储读文件 → FileStorage → FileService.upload_document → parse_file_by_sdk」为**通用函数**（如 `storage_to_rag(file_meta_list, user_info, doc_status)`），入参为统一结构的 file_meta_list（含 path、filename、content_type、kb_id、user_id 等）；feishu 与 kms 均调用该函数。 |
| **删除文档时清理 bucket** | 在 `document_app` 删除文档时，除 `delete_feishu_bucket_file` 外，增加 `delete_kms_bucket_file(doc_id)`（根据 doc_id 查 kms 表得到 oss_path，再删 MinIO/OSS）。若表结构统一为「source+doc_id」可统一一个 delete_xxx_bucket_file(source, doc_id)。 |
| **定时任务** | 在 `api/apps/__init__.py`：若「仅蓝凌」则关闭飞书调度（`init_scheduler` 不调或加开关）；新增蓝凌定时任务（如每日 6:00 全量/增量调用 kms_sync 的查询+下载+入 RAG）。若「飞书+蓝凌」并存，则两套任务都注册，用配置开关控制。 |
| **配置** | 在 `conf/service_conf.yaml` 或 feishu_sync 同套 configs 中增加 `kms_info`：base_url、auth_type、basic_user、basic_password、sm3_key、bucket_prefix 等；生产/测试环境可不同 base_url。 |

### 3.3 需要改动的现有文件（建议）

| 文件 | 改动要点 |
|------|----------|
| **api/apps/__init__.py** | 增加「数据源开关」或「同步源列表」；注册 kms_sync 蓝图；若仅蓝凌则不再调用 `start_scheduler_accord_time()`，或改为调用蓝凌定时任务。 |
| **api/apps/document_app.py** | 删除文档时在现有 `delete_feishu_bucket_file` 之后（或统一入口内）调用 `delete_kms_bucket_file(doc_id)`，避免蓝凌来源文件残留在 bucket。 |
| **feishu_sync/sync_operation.py**（可选） | 将 `oss_to_rag` 中「从 OSS 读 + 构建 file_meta + upload_document + 解析」抽到公共模块（如 `rag_sync_common.py`），供 feishu 与 kms 共用，减少重复。 |

### 3.4 不改动或仅配置改动的部分

- **FileService.upload_document、DocumentService、KnowledgebaseService、parse_file_by_sdk、delete_file_by_doc_id、delete_file_chunks_by_doc_id**：逻辑不变，仅调用方由 feishu 改为 kms 或共用封装。
- **富文本流程**：若蓝凌侧也有 doc/pdf/ppt 等，可继续走 `call_rich_text_backend`；若蓝凌只提供附件下载（如 PDF），则仅普通上传+解析即可。
- **前端**：若当前有「飞书同步」入口，需改为「知识库同步」或增加「蓝凌同步」入口；若仅后台定时同步、无前端选文件，则可能仅配置与开关即可。

## 四、工时评估（人天）

假设 1 人天 = 1 名开发 1 个工作日，环境为：蓝凌测试环境可用、认证方式为 Basic 或 SM3 二选一先实现一种。

| 序号 | 任务项 | 说明 | 人天 |

| 2 | 蓝凌 HTTP 客户端与认证 | 实现 kms_sync 的 config、auth（Basic + 国密 SM3）、client（queryKnowledge、getDonwloadUrl、下载文件流）；单测或脚本验证。 | 1.5 |

| 4 | 同步流程与 RAG 对接 | 实现 kms_sync/sync_operation：全量/增量/按 fdId 查询 → 附件下载 → 写 MinIO/OSS → 调用「storage 到 RAG」通用逻辑（上传+解析）；处理失效/退出（删或标记）。 | 2 |
| 5 | 通用「storage → RAG」抽离（可选） | 从 feishu sync_operation 抽离 oss_to_rag 核心为共用函数，feishu/kms 都调用；避免重复。 | 0.5 |

## 五、小结

- **飞书流程**：定时/手动 → 遍历飞书节点 → 飞书 API 导出/下载 → OSS/MinIO → upload_document + 解析 → 更新 feishu_documents_info。
- **蓝凌对接**：用「知识查询接口」替代遍历、「附件下载接口」替代飞书下载；之后同样写 MinIO/OSS，再走现有上传与解析；可选增加「变更通知」接收。
- **改动范围**：新增 `kms_sync/`（客户端、认证、同步逻辑、可选通知接口、表结构）；复用存储与 RAG 上传/解析，并抽离共用「storage → RAG」逻辑；调度与删除文档处增加蓝凌分支或开关。
- **工时**：约 **8～10 人天**（含联调与测试）；不做变更通知或仅 Basic 认证时可略减。

以上可作为开发排期与评审依据；具体表结构、接口路径和配置键名需与蓝凌/云从文档及现有项目命名保持一致。
