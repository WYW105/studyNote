# RAGFlow 线上资源与生产环境冲突排查报告

> 排查范围：可能浪费线上资源（如 DeepSeek API）、与生产环境冲突的配置与代码，包括源代码和数据库。

---

## 一、配置与密钥来源（谁在“用线上”）

### 1. 配置文件：`conf/service_conf.yaml` 与 `docker/service_conf.yaml.template`

- **`user_default_llm`**（默认被注释）  
  - 若取消注释并填写 `api_key`、`base_url`、`default_models`，则：
    - 会作为**默认 LLM 配置**被 `api/settings.py` 的 `init_settings()` 读取。
    - `API_KEY`、`LLM_FACTORY`、`CHAT_MDL`、`EMBEDDING_MDL` 等均来自此处。
  - **风险**：若在开发/测试环境写入真实 `api_key` 或生产 `base_url`，部署到生产或误用该配置会**直接调用线上 API 或生产端点**。
  - **建议**：生产与测试严格分离配置；不要在同一份 `service_conf.yaml` 里混用生产 key。

- 当前仓库中：
  - `conf/service_conf.yaml` 里 **未启用** `user_default_llm`（仅注释示例）。
  - `docker/service_conf.yaml.template` 中 `user_default_llm` 也为注释，**不会默认带出 API 调用**。

### 2. 数据库表：存的是“用哪个线上服务”

- **`tenant_llm`**（表名：`tenant_llm`）  
  - 字段：`tenant_id`, `llm_factory`, `llm_name`, **`api_key`**, **`api_base`** 等。  
  - 作用：每个租户绑定的 LLM/Embedding/Rerank 等模型的 **API Key 和 Base URL**。  
  - 实际调用 LLM/Embedding 时，`api/db/services/llm_service.py` 的 `get_model_config()` → `model_instance()` 会从这里取 `api_key` 和 `api_base` 去请求外部接口。  
  - **风险**：  
    - 若把**生产库**的库导出/恢复到测试环境，测试环境会拿生产的 key 去打生产 API，造成计费与数据混乱。  
    - 或测试环境库被误接到生产 RAGFlow，同样会打生产 API。  
  - **建议**：测试/预发/生产使用**不同库**，且测试库中 `tenant_llm.api_key` 使用测试用 key 或占位符。

- **`dialog` 的 `prompt_config`（JSON）**  
  - 其中可包含 **`tavily_api_key`**。  
  - 当 `api/db/services/dialog_service.py` 中 `prompt_config.get("tavily_api_key")` 为真时，会调用 **Tavily 搜索 API**（`rag/utils/tavily_conn.py`）。  
  - **风险**：对话若在测试环境复用了带真实 `tavily_api_key` 的配置，会消耗线上 Tavily 额度。  
  - **建议**：测试环境对话不要配置真实 Tavily key，或单独用测试 key。

- 其他与“线上”相关的多为业务数据（如知识库、文档），不直接存第三方 API key，但**租户默认模型 ID**（如 `tenant.llm_id`、`embd_id`）会指向 `tenant_llm`，从而间接决定是否用线上 API。

## 二、源代码中“会打线上”的路径（无 key 不请求）

以下路径都会**先取配置/DB 中的 key 或 base_url**，再发请求；没有配 key 时不会真正打第三方（或会报错）。

| 位置 | 行为 | 密钥/URL 来源 |
|------|------|----------------|
| `api/settings.py` | `API_KEY`、`LLM_FACTORY`、`CHAT_MDL`、`EMBEDDING_MDL` 等 | `conf/service_conf.yaml` 的 `user_default_llm` |
| `api/db/services/llm_service.py` | 所有 LLM/Embedding/Rerank/TTS/ASR 等模型实例化 | 表 `tenant_llm`（`api_key`、`api_base`） |
| `api/db/services/dialog_service.py` | 对话检索 + LLM 生成；可选 Tavily、知识图谱 | 对话的 `prompt_config.tavily_api_key`；LLM 仍来自 `tenant_llm` |
| `agentic_reasoning/deep_research.py` | Deep Research 推理中联网检索 | `prompt_config.tavily_api_key` |
| `agent/component/retrieval.py` | Agent 检索工具 | 组件参数 `tavily_api_key` |

**结论**：  
- 没有在 `service_conf.yaml` 或 DB 的 `tenant_llm` 里配置真实 key，**不会**调用 DeepSeek/OpenAI 等。  
- 没有在对话/Agent 里配置 `tavily_api_key`，**不会**调用 Tavily。

## 三、代码中的“默认 URL”（仅当选用该厂商时生效）

以下为各厂商客户端在代码里的**默认 base_url**，只有在用户选择了对应 factory 且未覆盖 `base_url` 时才会用到，**不会在未配置该厂商时主动请求**。

| 文件 | 默认 URL / 说明 |
|------|------------------|
| `rag/llm/chat_model.py` | DeepSeek: `https://api.deepseek.com/v1`；OpenAI: `https://api.openai.com/v1`；SiliconFlow: `https://api.siliconflow.cn/v1` |
| `rag/llm/embedding_model.py` | OpenAI: `https://api.openai.com/v1`；SiliconFlow: `https://api.siliconflow.cn/v1/embeddings` |
| `rag/llm/rerank_model.py` | Jina: `https://api.jina.ai/v1/rerank`；SiliconFlow: `https://api.siliconflow.cn/v1/rerank` 等 |
| `rag/llm/tts_model.py` | OpenAI: `https://api.openai.com/v1`；SiliconFlow TTS 等 |
| `rag/llm/sequence2txt_model.py` | OpenAI Whisper: `https://api.openai.com/v1` 等 |

**结论**：这些都是“该厂商的默认端点”，**只有在该租户配置了对应 factory + api_key 后**才会被使用，不会单独造成“误用线上”或与生产冲突；与生产冲突主要来自**同一 key/base_url 被用在错误环境**（见第一节）。

## 四、HuggingFace（模型下载，非按次计费 API）

- **使用处**：`rag/llm/embedding_model.py`、`rag/llm/rerank_model.py`、`deepdoc/vision/*.py`、`deepdoc/parser/pdf_parser.py`、`download_deps.py` 等通过 `huggingface_hub.snapshot_download` 拉取模型。
- **环境变量**：多处注释建议 `export HF_ENDPOINT=https://hf-mirror.com`，用于国内镜像，避免直连 HuggingFace。
- **与生产冲突**：主要是网络/镜像配置问题，一般不会“误扣生产 API 费用”；若生产也走同一镜像，注意带宽与缓存策略即可。

## 五、初始化时是否会自动调 LLM/Embedding（易误以为“自动用线上”）

- **`api/db/init_data.py`**  
  - `init_web_data()` 中 **未调用** `init_superuser()`（已被注释）。  
  - 因此**正常启动不会**执行 `init_superuser()` 里那一次“Hello!”聊天和 Embedding 测试。  
- **若将来启用** `init_superuser()`（例如首次安装时创建管理员）：  
  - 会使用当前 `settings.CHAT_MDL`、`settings.EMBEDDING_MDL` 和 `settings.API_KEY`（来自 `user_default_llm`）各请求一次。  
  - 若此时 `service_conf.yaml` 里配置了真实 key，**会真的打一次线上 API**。  
- **建议**：若启用 `init_superuser()`，确保对应环境（如测试）的 `user_default_llm` 使用测试 key 或留空，避免用生产 key。

## 六、测试/示例中的硬编码（易误连生产或浪费额度）

| 位置 | 内容 | 风险 |
|------|------|------|
| `sdk/python/test/test_sdk_api/t_session.py`、`t_agent.py` 等 | 硬编码 `API_KEY = "ragflow-BkOGNhYjIyN2JiODExZWY5MzVhMDI0Mm"` | 若测试指向**生产 RAGFlow 地址**（如通过 `HOST_ADDRESS`），会用该 token 访问生产，可能改生产数据或触发生产侧调用 LLM。 |
| `sdk/python/test/conftest.py` | 使用 `HOST_ADDRESS = os.getenv('HOST_ADDRESS', 'http://127.0.0.1:9380')`，API Key 通过登录后 `new_token` 获取 | 默认本机；若 CI/本机把 `HOST_ADDRESS` 指到生产，则同上。 |
| `example/sdk/dataset_example.py` | `API_KEY = "ragflow-IzZmY1MGVhYTBhMjExZWZiYTdjMDI0Mm"`，`HOST_ADDRESS = "http://127.0.0.1"` | 同上：若有人把 base_url 改成生产地址，会以该 key 访问生产。 |

**建议**：  
- 测试/示例统一使用环境变量（如 `RAGFLOW_API_KEY`、`RAGFLOW_BASE_URL`），且 CI 与本地**禁止**将 `HOST_ADDRESS` 指到生产。  
- 示例中的 key 明确标注为“示例用，请替换为本地或测试 key”。

## 七、汇总：如何避免浪费线上资源、避免与生产冲突

1. **配置**  
   - 生产与测试使用**不同** `service_conf.yaml`（或通过环境变量/密钥管理注入），**不要**在测试配置里写生产 `api_key` 或生产 `base_url`。  
   - 保持 `user_default_llm` 在未明确需要的环境中注释掉，或仅填测试用 key。

2. **数据库**  
   - 测试/预发/生产使用**不同库**；测试库中 `tenant_llm` 不要使用生产 API key。  
   - 避免把生产库直接恢复到测试 RAGFlow 实例。

3. **对话/Agent**  
   - 测试环境不在 `prompt_config` 或 Agent 参数中配置生产用 `tavily_api_key`；可留空或使用测试 key。

4. **测试与示例**  
   - 不在代码中写死生产 API Key；不用 `HOST_ADDRESS` 指向生产。  
   - 示例与 SDK 测试建议改为从环境变量读取 key 和 base_url，并在文档中说明。

5. **初始化**  
   - 若将来启用 `init_superuser()`，确保该环境下的 `user_default_llm` 与目标环境一致（测试用测试 key，生产用生产 key），避免一次初始化就打到生产 API。

按上述方式隔离配置与数据后，不会出现“代码或库默认在浪费 DeepSeek 等线上资源”或“测试环境误打生产 API”的问题；真正产生调用的，都是**当前环境显式配置的 key/URL**。
